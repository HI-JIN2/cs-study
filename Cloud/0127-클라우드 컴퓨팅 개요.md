# Cloud Computing 개요 (0127)

## 1. Cloud Computing vs On-Premise

### 1) On-Premise
기업이 자체 데이터 및 솔루션 등을 저장하기 위해 데이터 센터를 구축해서 IT 서비스를 수행하는 방식입니다.

*   **특징**:
    *   하드웨어를 포함한 모든 자원(CPU, 메모리, 디스크, OS, 네트워크, 라이선스 등)에 대한 초기 투자 비용이 큽니다.
    *   탄력적이지 않은 제한된 용량으로 인해 지속적 관리 비용이 증가할 수 있습니다.
    *   기업에 내재화된 서비스를 통해 품질 및 보안에 대한 신뢰도는 높습니다.
    *   **클라우드 전환 이유**: 높은 초기 도입 비용과 운용에 따른 추가 비용 부담 때문입니다.
    *   **설계 방식**: 자원 사용량은 최대 사용량을 근거로, 네트워크 트래픽은 최대 순간 트래픽을 가정하여 고사양 설계를 하게 되며, 증설에 따른 시간적/인적 비용이 발생합니다.
    *   **비용 측면**: 1~5년 간의 고정적 비용을 고려했을 때, 특정 시점부터는 Cloud 비용이 더 커질 수도 있으므로, Cloud 도입은 비용 절감보다는 **서비스의 가용성과 품질** 향상에 초점을 맞춰야 합니다.

### 2) Cloud Computing
정보 처리를 자신이 보유한 컴퓨터가 아닌 인터넷 너머에 존재하는 컴퓨터에서 처리하는 사고방식 또는 개념입니다.

*   **핵심**: IT 자산을 소유하는 것이 아니라 **서비스로 이용**하는 모델입니다.
*   **이용 방식**: 인터넷에 접속해서 웹 브라우저나 Cloud 전용 소프트웨어를 통해 서비스를 이용합니다.
*   **장점**: 서비스 제공자의 최소한의 관리나 개입만으로 신속하게 생성/제거/구성할 수 있으며, 언제 어디서든 필요할 때 네트워크에 접속하여 컴퓨팅 자원을 공유하여 사용할 수 있습니다.

## 2. Cloud Computing의 등장 배경 및 정의

### 1) 등장 배경
*   **1980년대 (메인 프레임 시대)**: 데이터와 애플리케이션을 메인 프레임에 모아서 처리. 단말기는 입출력 기능만 존재.
*   **1990년대 (클라이언트-서버 모델)**: 단말기에도 처리 기능을 부여한 분산형 모델이 주류.
*   **2000년대 (웹 기반 시스템)**: 사내 시스템이 네트워크 환경 위에 구축되며 서버가 처리를 담당.
*   **2010년대 (Cloud Computing)**: 분산 배치된 서버 리소스를 필요할 때 필요한 만큼 사용하는 모델 등장.

*   **보급 배경**:
    *   **기술 발전**: CPU 처리 속도 고속화, 가상화 및 분산 처리 기술 발전.
    *   **환경 변화**: 모바일의 융성, 빠르고 저렴해진 네트워크, 거대 데이터 센터에 의한 규모의 경제 실현.
    *   **인식 변화**: 기업 사용자와 Cloud 사업자 모두에게 Cloud를 받아들일 환경이 조성됨.

### 2) 정의
공유 구성이 가능한 컴퓨팅 리소스의 통합을 통해, 어디서나 간편하게 요청에 따라 네트워크를 통해 접근하는 것을 가능하게 하는 모델입니다. 최소한의 이용 절차 또는 서비스 공급자의 상호 작용을 통해 신속히 할당되어 제공됩니다.

### 3) 특징 (NIST 정의)
1.  **On-Demand Self-Service**: 사용자가 별도의 기술 습득 없이, 필요할 때 온라인으로 즉시 자원을 확보하여 사용 가능.
2.  **Broad Network Access**: 시간, 장소, 단말(Any time, Any Place, Any Device)에 구애받지 않고 접근 가능.
3.  **Resource Pooling**: 다중 임대 모델(Multi-tenancy)을 퉁한 자원 할당 및 공유.
4.  **Rapid Elasticity**: 비즈니스 상황에 따른 컴퓨팅 자원의 탄력적 사용 (Flexibility, Scalability).
5.  **Measured Service**: 사용한 만큼 비용을 지불 (Pay-Per-Use, Pay as you go).

### 4) 장점
*   **경제성**: 사용한 만큼만 지불.
*   **자동화**: 업데이트, 보안 패치, 백업 등을 직접 수행할 필요 없음.
*   **이동성**: 웹을 지원하는 모든 장치에서 접근 가능.
*   **자원 공유**: 효율적인 자원 활용.
*   **가용성**: 언제든지 서비스 이용 가능.
*   **빠른 구축**: 신속한 서비스 런칭 가능.
*   **확장성**:
    *   **Scale-Up**: 하드웨어 사양 업그레이드.
    *   **Scale-Out**: 서버 개수를 늘려 처리 능력 향상.
    *   *예시 (인디스쿨)*: 방학 중 트래픽 감소에도 고정 비용이 지출되던 문제를 클라우드 도입으로 해결. 트래픽 급증 시 유연한 대응 가능.
*   **중복성 및 탄력성 (Redundancy or Resilience)**: 데이터 센터(AZ 등) 분산을 통한 데이터 중복 저장으로 안정성 확보.

### 5) 한계
*   Internet Access 필수.
*   Security 및 Privacy 우려.
*   **Vendor Lock-In**: 특정 클라우드 벤더에 종속될 수 있음.

## 3. 클라우드 서비스 종류

### 1) 컴퓨팅 자원 계층
`Network` -> `스토리지` -> `서버` -> `가상화` -> `OS` -> `미들웨어` -> `런타임` -> `데이터` -> `애플리케이션`

### 2) 서비스 모델 비교

#### On-Premise
모든 자원(인프라부터 애플리케이션까지)을 직접 구축하고 관리하는 방식.

#### IaaS (Infrastructure as a Service)
*   서버, 스토리지, 네트워크 등 인프라 자원을 가상화하여 제공하는 서비스.
*   **특징**: 자동화된 확장성, 사용량 기반 과금.
*   **관리 범위**: 인프라 자원(CPU, 메모리, 디스크 등)의 특징에 대한 지식이 필요하며, OS 및 애플리케이션은 사용자가 직접 관리.
*   **예시**: AWS EC2, and all CSPs provide IaaS.

#### PaaS (Platform as a Service)
*   애플리케이션 개발, 실행, 관리를 위한 플랫폼(Runtime) 또는 프레임워크를 제공하는 서비스.
*   **특징**: 개발/실행 환경, DB 등이 미리 설치되어 있어 인프라 구축 없이 바로 개발 가능. 높은 생산성.
*   **차이점**: 서버, 네트워크, 보안 등을 Cloud 사업자가 관리. IaaS보다 관리가 쉽고, SaaS보다 자유도가 높음(직접 개발한 프로그램 가동).
*   **예시**: Salesforce force.com, Google App Engine, Heroku, Cloud Foundry, DB on Cloud (OCI).
    *   개발 프레임워크: Spring, Node.js, Ruby on Rails 등.
    *   메시지 브로커: Amazon SQS, RabbitMQ, Kafka 등.

#### SaaS (Software as a Service)
*   업무에 필요한 소프트웨어를 인터넷을 통해 제공받는 형태.
*   **특징**: 멀티테넌트 방식이지만 데이터는 사용자별로 분리되어 보안성 확보. 최신 기능 자동 업데이트.
*   **예시**: Google Apps, Office 365, Salesforce CRM, Naver Office.

#### DaaS (Desktop as a Service)
*   인터넷만 연결되면 언제 어디서나 기업 내부 망에 접속할 수 있는 가상 데스크톱 서비스.
*   **특징**:
    *   중앙 서버에 정보 저장 -> 보안성 강화 (단말기 분실/해킹에도 안전).
    *   **연속성**: 장소/기기에 상관없이 동일한 업무 환경 제공.
    *   **비용 절감**: 초기 구축 비용 없음, 구독 형태.
    *   **VDI vs DaaS**: 가상화 인프라를 직접 보유하면 VDI, Cloud 사업자가 제공하면 DaaS.
    *   **보안성**: 화면만 전송하는 기술로 개인 PC 보안 취약점에 영향받지 않음.

## 4. 클라우드 이용 모델 (Deployment Models)

### 1) Public Cloud
불특정 다수에게 서비스 제공 (AWS, GCP, Azure 등).
*   사용량 기반 요금제.
*   사용자 간 간섭 없음 (논리적 격리).
*   IT 자산 미보유로 인한 저비용, 관리 부담 감소.

### 2) Private Cloud
특정 기업/사용자만을 위한 폐쇄적 환경.
*   자원과 데이터 제어권을 기업이 소유 -> 높은 물리적 보안성.
*   성능과 비용 최적화 가능.
*   **구현 방식**:
    *   **On-Premise Private Cloud**: 자사가 직접 보유 및 운영.
    *   **Hosted Private Cloud**: Cloud 사업자가 장비를 보유하고 전용 환경 제공.

### 3) Community Cloud
공통의 목적을 가진 특정 기업들이 공동으로 운영하는 형태 (Public과 Private의 중간).

### 4) Hybrid Cloud
Public Cloud와 Private Cloud를 결합.
*   데이터/애플리케이션의 유연한 이동 가능.
*   **활용**: 중요 데이터/핵심 업무는 Private Cloud, 트래픽 변동이 심하거나 덜 중요한 워크로드는 Public Cloud 사용.

### 5) Multi Cloud
2개 이상의 서로 다른 클라우드 제공업체 서비스를 조합하여 사용.
*   **장점**: 특정 벤더 종속(Vendor Lock-In) 최소화, 비즈니스 요구에 맞는 최적의 기능 선택 가능.

## 5. Cloud vs On-Premise 비교 및 핵심 기술

### 1) 비용 비교
*   **On-Premise**: 초기 투자 비용 높음. 장기간(5년 이상) 대규모 시스템 운용 시에는 더 저렴할 수도 있음.
*   **Cloud**: 초기 비용 낮음. 기존 시스템 수정/데이터 이행 비용(Migration Cost)이 발생할 수 있음.

### 2) 안정성 및 신뢰성
*   **Cloud 리스크**: 사업자의 하드웨어 장애, 서비스 중단, 통신 위협 등.
*   **보안 거버넌스**: 데이터 처리를 사업자에게 위탁하므로 통제권 문제 발생 가능.
*   **대책**: 가상 서버 백업, 템플릿 마련, 벤더 신뢰성 검토 필요.

### 3) 클라우드 핵심 기술 2가지

#### 가상화 (Virtualization)
*   하드웨어(물리적 자원)를 소프트웨어로 대체하는 기술.
*   **가상 서버**: 물리 서버 1대 위에 여러 게스트 서버를 생성.
*   **특징**:
    *   물리적 자원을 가상으로 분할/통합 가능.
    *   메모리/스토리지 용량 조절 유연 (Scale-Up/Down).
    *   단, 가상 서버 자체 성능 향상(Scale-Up)에는 한계가 있어 서버 대수를 늘리는(Scale-Out) 것이 유리함.

#### 분산 처리와 로드밸런서 (Distributed Processing & Load Balancer)
*   트래픽을 여러 서버로 분산시켜 시스템 전체 성능과 가용성을 높이는 기술.
