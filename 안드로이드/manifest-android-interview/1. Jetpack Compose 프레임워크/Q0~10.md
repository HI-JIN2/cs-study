Compose Compiler, Compose Runtime, Compose UI


Jetpack Compose는 UI를 화면에 그릴 때 Composition, Layout, Drawing
1. composition: @Composable 함수를 실행하고 UI 트리를 구축
2. Layout: 제공된 제약 조건에 따라 각 UI 컴포넌트의 크기와 위치를 결정



기존의 view 시스템은 상속을 기반으로함
compose는 컴포지션(조합)을 기반으로 함

상속을 피해야하는 이유
재사용이 어렵고 복잡성이 높음
커스텀 ui 구성이 어려움

- 컴포지션: 처음 UI를 구성하는 과정
- 리컴포지션(Recomposition): 상태가 바뀌면 필요한 부분만 다시 그림
	- 리컴포지션 되는 범위를 작게 만들면 좋음
	- 컴포저블이 상태를 가지는 것보다 위에서 내려받게끔 구현 (컴포저블은 stateless)
		```kotlin
		@Composable
		fun Counter(
		    count: Int,
		    onIncrease: () -> Unit
		)
		```

### Q1. 왜 Compose는 상속보다 컴포지션을 쓰나요?

**모범 답변**

> 상속은 변경에 취약하고 재사용이 어렵기 때문에, Compose에서는 작은 Composable들을 조합하는 컴포지션 방식을 사용합니다.

---

### Q2. 컴포지션이 성능에 좋은 이유는?

> UI가 작은 단위로 나뉘어 있어, 상태 변경 시 전체가 아닌 필요한 부분만 리컴포지션되기 때문입니다.

---

### Q3. 컴포지션이 잘못되면 어떤 문제가 생기나요?

- 불필요한 recomposition
    
- 성능 저하
    
- 가독성 저하
    

**추가 멘트**

> 그래서 remember, key, derivedStateOf 등을 적절히 사용합니다.

---

## 8. 한 문장 요약 (면접 마무리용)

> Jetpack Compose는 UI를 상속이 아닌 컴포지션 방식으로 구성하며, 작은 Composable들을 조합해 재사용성과 유지보수성을 높이고, 상태 변경 시 필요한 부분만 리컴포지션되도록 설계되었습니다.
